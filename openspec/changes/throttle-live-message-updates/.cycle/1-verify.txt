### Issues Found

WARNING: flux/kawa/src/markdown-to-simplex.ts:42 ÔÇö SimpleX bold `*text*` is converted to `_text_` (italic) instead of being preserved unchanged. Spec requires: "WHEN extractMessageText receives text containing `*correct bold*`, THEN the output SHALL contain `*correct bold*` unchanged". This creates a conflict with the SYSTEM.md prompt (which tells the LLM to use `*bold*` for SimpleX bold) ÔÇö the converter then breaks that output by making it italic. The `**bold**` Ôćĺ `*bold*` path works correctly, so standard markdown bold functions properly, but the two defense layers (system prompt + converter) work against each other when the LLM follows the system prompt.

WARNING: flux/kawa/tests/markdown-to-simplex.test.ts:107 ÔÇö Test "preserves already-correct SimpleX bold *text*" has no assertions (empty body with only an explanatory comment). This masks the spec violation above. The test passes vacuously instead of asserting the required behavior, which would fail: `convertMarkdownToSimplex("*bold*")` Ôćĺ `"_bold_"` rather than the spec-required `"*bold*"`.

WARNING: flux/kawa/src/live-message-throttler.ts:37 ÔÇö `flush()` sends a `sender.updateLiveMessage()` call even when no throttle timer is pending, violating the spec: "Flush with no pending timer ÔÇö the method SHALL be a no-op (no update sent if no buffered text has changed since last send)". When called from tool_execution handlers with no pending timer, this produces one redundant SimpleX command. Not harmful to UX (same text content), but a spec violation that could be fixed by tracking whether text has changed since the last send.

SUGGESTION: flux/kawa/src/kawa.ts:564-566 ÔÇö `handleIncomingMessage` manually clears `ctx.throttleTimer` instead of calling `throttler.cancel(ctx)`, even though the throttler instance is available as a parameter. This duplicates the cancel logic and would miss any future additions to `cancel()` (e.g., logging, metrics).

SUGGESTION: flux/kawa/src/session-manager.ts:87-92 ÔÇö `SessionManager.closeAll()` unsubscribes sessions but does not cancel throttle timers, unlike `removeByContactId()` which does. While harmless at process shutdown (timers are GC'd on exit), an orderly shutdown calling `closeAll()` alone could leave dangling timers if the process doesn't immediately exit.

SUGGESTION: openspec/changes/throttle-live-message-updates/specs/kawa-message-sender/spec.md ÔÇö Spec references `sender.updateMessage()` but the code uses `sender.updateLiveMessage()`. Minor spec typo.
