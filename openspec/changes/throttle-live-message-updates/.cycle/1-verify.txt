### Issues Found

WARNING: openspec/changes/throttle-live-message-updates/specs/kawa-message-sender/spec.md:22 — Method name typo: spec says `sender.updateMessage(ctx, ctx.accumulatedText)` but the actual code uses `sender.updateLiveMessage(ctx, textToSend)`. The method `updateMessage` does not exist; it should be `updateLiveMessage`.

WARNING: flux/kawa/src/message-sender.ts:35 — `updateLiveMessage(ctx: ContactContext, text: string)` accepts a `text` parameter but ignores it, always reading `ctx.accumulatedText` directly. Callers (throttler, kawa.ts) always pass `ctx.accumulatedText` as the argument, so behavior is correct, but the unused parameter is misleading and could cause future bugs if a caller passes a different value expecting it to be used.

WARNING: flux/kawa/src/live-message-throttler.ts:38-44 — `scheduleUpdate`'s timer callback sets `ctx.lastSentText = textToSend` in the `.then()` handler without checking `ctx.generation` for staleness. In contrast, `flush()` has a `if (ctx.generation !== gen) return;` guard before setting `lastSentText`. This inconsistency means a stale `lastSentText` value could be written if a generation change occurs during the async `updateLiveMessage` call. Mitigated by `handleIncomingMessage` resetting `lastSentText = ""` and `cancel()` clearing timers, but inconsistent with the safety pattern established in `flush()`.

SUGGESTION: The `lastSentText: string` field on `ContactContext` is used for the "skip if text hasn't changed since last send" no-op optimization mentioned in the live-message-throttle spec ("Flush with no pending timer" scenario), but it is not explicitly called out in the tasks (task 1.8 only mentions `throttleTimer`). Consider documenting it for completeness.
