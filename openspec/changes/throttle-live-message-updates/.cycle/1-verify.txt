### Issues

WARNING: flux/kawa/src/live-message-throttler.ts:60 — `flush()` sends an `updateLiveMessage` call even when no throttle timer is pending and no text has changed since the last send. Spec scenario "Flush with no pending timer" says: "the method SHALL be a no-op (no update sent if no buffered text has changed since last send)". In practice this causes a redundant idempotent update call when `flush()` is called before tool markers or agent_end with no pending throttled update — harmless but technically non-compliant with spec.

WARNING: flux/kawa/src/markdown-to-simplex.ts:61 — `*text*` (single asterisk) is always converted to `_text_` (italic), even when it represents SimpleX-native bold. Spec requirement "Preserving SimpleX-native formatting" says: "WHEN extractMessageText receives text containing `*correct bold*`, THEN the output SHALL contain `*correct bold*` unchanged". The regex approach cannot distinguish SimpleX-native `*bold*` from standard `*italic*`. This creates a conflict with the `.pi/SYSTEM.md` which tells the LLM to use `*bold*` for bold — if the LLM follows SYSTEM.md, the converter will incorrectly turn it into `_italic_`. The unit test documents this known limitation. To resolve: either update the spec to reflect this tradeoff, or add tracking state to detect whether `*text*` was produced by the LLM vs converted from `**text**`.

WARNING: flux/kawa/src/live-message-throttler.ts:73-76 — `LiveMessageThrottler.cancel()` method exists but is never called. Timer cleanup is done inline in `session-manager.ts:60-62`, `kawa.ts:564-566`, and `commands.ts:63-65`. Task 1.10 says "call throttler's `cancel()`" and task 1.11 implies the same. The behavior is functionally equivalent, but the uncalled `cancel()` method is dead code.

SUGGESTION: flux/kawa/src/live-message-throttler.ts:34-38 — The `scheduleUpdate` timer callback does not include generation-based staleness detection. The design doc's "Risks/Trade-offs" section mentions "Timer callback checks `ctx.throttleTimer === null` to detect cancellation and skips the update" as a mitigation, but the implementation doesn't perform this check — it relies on `clearTimeout` being called before the generation changes. This is functionally correct (JavaScript's single-threaded event loop prevents the race), but diverges from the stated design. Consider adding `const gen = ctx.generation` before the `updateLiveMessage` call and checking after, for symmetry with `flush()`.
