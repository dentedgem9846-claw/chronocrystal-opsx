### Issues

WARNING: flux/kawa/src/live-message-throttler.ts:30-45 — `lastSentText` set in `.then()` callback even when `updateLiveMessage` skips the actual send (because `liveMessageItemId === null`). This happens when a throttle timer fires before `startLiveMessage` resolves. The `.then()` resolves successfully (since `updateLiveMessage` returns `Promise<void>` after logging a warning) and sets `lastSentText` to a text value that was never sent to SimpleX. This could cause the no-op flush optimization to incorrectly skip a legitimate update. The impact is limited: `finalizeLiveMessage` always sends regardless of `lastSentText`, so the final message is always correct. At worst, intermediate streaming display may momentarily lag. Recommendation: Either (a) have `updateLiveMessage` return a boolean/sentinel indicating whether the update was actually sent, so `.then()` can conditionally set `lastSentText`, or (b) move `lastSentText` reset into the timer callback only after verifying `ctx.liveMessageItemId !== null`.

SUGGESTION: flux/kawa/src/kawa.ts:292 — `KAWA_LIVE_MSG_UPDATE_INTERVAL_MS` env var parsing uses `Number(process.env.KAWA_LIVE_MSG_UPDATE_INTERVAL_MS ?? default)`. If set to `""` (empty string), `Number("")` returns `0`, making `setTimeout(cb, 0)` defer to the next tick rather than bypassing the throttle entirely. Consider adding a validation check: `if (isNaN(val) || val < 0) { console.warn(...); val = defaultConfig.liveMessageUpdateIntervalMs; }`. Also consider whether `interval=0` should bypass the throttler completely (direct call to `sender.updateLiveMessage`).

SUGGESTION: The known issue in `issues.md` about `MessageSender.updateLiveMessage` ignoring its `text` parameter appears to be fixed in the current code — the `text` parameter IS passed through to `updateLiveMessageCmd`. However, every caller passes `ctx.accumulatedText` as the argument, making the parameter semantically redundant. Consider either removing the `text` parameter and reading `ctx.accumulatedText` internally (simpler), or documenting that the parameter allows callers to specify text that differs from context state (more explicit). This is a code clarity concern, not a correctness bug.
