### WARNING

**W1: `finalizeLiveMessage` does not transition state to IDLE when `liveMessageItemId` is null**

The spec scenario "Finalize when live message item ID is null" states:
> **AND** the contact state transitions from STREAMING to IDLE

But `message-sender.ts:105-114` only transitions state inside the `if` block that requires `ctx.liveMessageItemId !== null`:

```ts
if (ctx.liveMessageState === "STREAMING" && ctx.liveMessageItemId !== null) {
    await this.updateLiveMessageCmd(...);
    ctx.liveMessageState = "IDLE";   // ← only reached when itemId is non-null
    ctx.liveMessageItemId = null;
    ctx.accumulatedText = "";
}
```

When `startLiveMessage` fails (returns null), the caller still sets `ctx.liveMessageState = "STREAMING"` (`kawa.ts:189`), but `ctx.liveMessageItemId` stays null. When `agent_end` later calls `finalizeLiveMessage(ctx)`, the state **does not transition back to IDLE**, leaving the contact stuck in STREAMING state permanently. The error handler in `wireSessionEvents` does reset state on thrown errors, but `agent_end` is not an error event.

**Recommendation**: Restructure `finalizeLiveMessage` to always transition to IDLE, moving state reset outside the null guard:
```ts
async finalizeLiveMessage(ctx: ContactContext): Promise<void> {
    if (ctx.liveMessageState === "STREAMING") {
        if (ctx.liveMessageItemId !== null) {
            await this.updateLiveMessageCmd(..., false);
        }
        ctx.liveMessageState = "IDLE";
        ctx.liveMessageItemId = null;
        ctx.accumulatedText = "";
    }
}
```
Reference: `message-sender.ts:105-114`, spec scenario "Finalize when live message item ID is null"

### SUGGESTION

**S1: `updateLiveMessageCmd` is public but intended as internal**

`updateLiveMessageCmd` (`message-sender.ts:82`) has no access modifier, defaulting to public. The spec requirement "No send-or-update indirection" states: *"Callers SHALL use the public `updateLiveMessage` method"*, implying `updateLiveMessageCmd` is not part of the caller-facing API. Currently no external code calls it directly (confirmed via grep), so there is no safety issue today — but marking it `private` would enforce the intended boundary.

This is already tracked as ISSUE-002 in `issues.md`.

**Recommendation**: Add `private` keyword to `updateLiveMessageCmd` at `message-sender.ts:82`. One-line change.
