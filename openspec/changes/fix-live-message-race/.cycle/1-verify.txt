### Issues

#### ⚠️ WARNING (Should fix)

**1. Error handler in `wireSessionEvents` bypasses generation check**

- **Location:** `kawa.ts:157-162`
- **Detail:** The `.catch()` handler resets `ctx.liveMessageState`, `ctx.liveMessageItemId`, and `ctx.accumulatedText` without checking `ctx.generation`. If an in-flight event from a prior prompt throws during an await (e.g. `startLiveMessage` rejects), and the user has already sent a new message (incrementing generation), the error handler would overwrite the new prompt's state on the same `ContactContext` object.
- **Scenario:** (1) Event A's `startLiveMessage` throws → (2) User sends new message during the throw, `generation++` → (3) Error handler fires, blanks `liveMessageItemId` and `accumulatedText` for the current session
- **Recommendation:** Capture `const gen = ctx.generation` in the `wireSessionEvents` closure and check it before resetting state in the `.catch()` handler:
  ```ts
  handleAgentEvent(ctx, event, sender, formatter).catch((err) => {
      console.error(`[kawa] Error handling agent event:`, err);
      if (ctx.generation !== gen) return; // stale error, don't corrupt current state
      ctx.liveMessageState = "IDLE";
      ctx.liveMessageItemId = null;
      ctx.accumulatedText = "";
  });
  ```
  Note: for `/new`, the old context object is detached from the session manager, so the error handler's writes would be to an orphaned object (harmless). The risk is specific to `handleIncomingMessage` which reuses the same context.

#### 💡 SUGGESTION (Nice to fix)

**1. Redundant `ctx.accumulatedText = text` in `updateLiveMessage`**

- **Location:** `message-sender.ts:36`
- **Detail:** `updateLiveMessage` writes `ctx.accumulatedText = text`, but all callers in `handleAgentEvent` already set `ctx.accumulatedText` synchronously before calling the method (per D1). The redundant write is harmless but breaks the "caller owns synchronous state, method owns I/O" discipline that the design establishes.
- **Recommendation:** Consider removing the `ctx.accumulatedText = text` line from `message-sender.ts:36` since the caller is responsible for synchronous state mutation. Alternatively, add a comment acknowledging the redundancy as intentional safety.

**2. `agent_end` side effects execute before generation check**

- **Location:** `kawa.ts:231-235`
- **Detail:** In the `agent_end` case, `sendTextMessage` (line 231) or `finalizeLiveMessage` (line 234) are called before the generation check. If generation changes during the await, the I/O has already happened. The spec only requires that *state mutations* be prevented on stale events (which is satisfied), but a stale `agent_end` could potentially send a "(Agent finished with no output)" fallback message to a user who has already started a new conversation.
- **Recommendation:** Low priority — this is a minor UX edge case (fallback message in new session) rather than a state corruption bug. If desired, add a generation check before the I/O calls:
  ```ts
  if (ctx.generation !== gen) return; // check BEFORE sending
  await sender.sendTextMessage(ctx.contactId, "(Agent finished with no output)");
  ```
