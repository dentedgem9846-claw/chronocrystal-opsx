CRITICAL: flux/kawa/src/file-receiver.ts:134 — File path mismatch: `handleRcvFileComplete` constructs paths as `join(config.filesDir, subDir, pending.fileName)` (e.g., `kawa-files/images/photo.jpg`), but SimpleX's `setFilesFolder` command is configured to save received files flat in `config.filesDir` (e.g., `kawa-files/photo.jpg`). The `rcvFileComplete` event could provide the actual file path via `chatItem.chatItem.file.fileSource?.filePath`, but instead the code constructs an incorrect subdirectory path. This means all image, video, and generic file processing will fail at runtime because files won't be found at the expected paths. The `mkdirSync` calls in `kawa.ts` create subdirectories (`images/`, `videos/`, `files/`) that are never populated by SimpleX.

WARNING: flux/kawa/src/message-sender.ts:137-147 — `sendImageMessage` spec says "the file SHALL be sent using the `fileSource` field on `ComposedMessage` for local file reference", but the code sends images as inline base64 only (`msgContent: { type: "image", text, image: base64 }`) without `fileSource`. `sendFileMessage` correctly uses `fileSource: { filePath }`, so this is inconsistent.

WARNING: flux/kawa/src/kawa.ts:644-659 — Missing `rcvFileCancelled` event handling. The spec (file-receiver, "File transfer lifecycle event handling") says "Kawa receives a `rcvFileCancelled` or `rcvFileSndCancelled` event". The SimpleX SDK only defines `rcvFileSndCancelled` as a `ChatEvent` type — no separate `rcvFileCancelled` event exists. The code handles `rcvFileSndCancelled` correctly. However, a receiver-initiated cancellation would change the file status to `rcvCancelled` (via `CIFileStatus.RcvCancelled`) in a `newChatItems` update, which the code does not handle. If the receiver cancels, the pending buffer would never be resolved (timeout-only).

WARNING: flux/kawa/src/file-receiver.ts:268-272 — `MsgContent.Image` and `MsgContent.Video` in the SimpleX SDK types (`@simplex-chat/types`) do not define a `text` field (`MsgContent.Image` only has `{ type: "image" }`, `MsgContent.Video` only has `{ type: "video"; duration?: number }`). The code accesses `.text` via type casts (`(msgContent as MsgContent.Image).text`), which is a type safety violation. At runtime, if the SimpleX protocol doesn't include a `text` property on image/video content, this would always produce the fallback strings ("📷 Image" / "🎥 Video") instead of actual captions. The `MsgContent.File` type does have `text: string` and `MsgContent.Text` also has `text: string`, which is correct.
